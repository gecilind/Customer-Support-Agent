## 2. Layer Descriptions

| Layer | What It Is | What It Does NOT Do |
|---|---|---|
| **main.py** | Creates the app, initializes SQLAlchemy engine (Supabase) and httpx client at startup, registers middleware and routers, disposes on shutdown | No business logic, no routes, no SQL |
| **config.py** | Reads all env vars via pydantic-settings, validates at startup — missing vars crash the app immediately with a clear error | No logic — pure data definition |
| **core/exceptions.py** | Defines named errors: `AppError` (base), `ZendeskAPIError`, `KBSearchError`, `AIServiceError`, `IngestionError` | Does not catch errors — only defines them |
| **middleware/** | `request_handler.py` logs every request in/out and catches exceptions that escape controllers. `cors.py` adds browser headers for cross-domain access | No business logic. No layer calls middleware — FastAPI runs it automatically |
| **models/** | SQLAlchemy declarative models that mirror our DB tables. Alembic reads these to auto-generate migrations | Never used outside repositories. Not used for Infleet's DB (we don't own their schema) |
| **schemas/** | Pydantic data shapes that flow between layers — what the API receives and returns | No logic. They don't call anyone — everyone uses them |
| **db/** | SQLAlchemy async engine + session factory for Supabase (ours). Alembic migrations in `migrations/` | Does not execute queries — repositories do that |
| **repositories/** | Each repository knows exactly one data source and how to talk to it | Never contains business logic, never decides WHAT to do |
| **services/** | Makes decisions. Orchestrates KB search, OpenAI Chat Completions streaming, ticket detection, and DB persistence | Never touches HTTP, never knows about status codes |
| **controllers/** | Validates input, calls the service, catches named exceptions, returns HTTP status codes | Never contains business logic, never calls repositories directly |
| **routers/** | Maps a URL to a controller method. One line per route | Zero logic — no validation, no error handling |
| **dependencies.py** | Builds the full chain: repos → services → controllers. FastAPI's `Depends()` triggers the factories | Only imported by routers |

## 3. Communication Rules

### Call Direction

Strictly one-way downward. No layer ever calls upward or sideways.

```
Router → Controller → Service → Repository
```

### What Crosses Boundaries

**Schemas** cross every boundary. They are the shared language. Every layer reads and writes schemas.

**Models** never cross the repository boundary. The repository maps database rows to models internally, converts models to schemas, and returns schemas upward. No layer above the repository ever sees a model.

**Exceptions** flow upward. Repositories raise them. Services ignore them (let them pass through). Controllers catch them.

**DB connections flow downward.** `main.py` creates the SQLAlchemy engine → `dependencies.py` provides `AsyncSession` (Supabase) to repositories → repositories use them to execute queries. No other layer touches a session or pool.

### Who Calls Who — Complete Map

```
main.py
├── registers middleware/ (cors.py, request_handler.py)
├── creates db/ pools → stores on app.state
└── mounts routers/

Router
└── calls Controller (via dependencies.py)

Controller
├── validates schemas/
├── calls Service
└── catches core/exceptions.py

Service
├── calls OpenAI Chat Completions API (gpt-4o-mini) directly
├── calls Repositories
└── uses schemas/ only (never models/)

Repository
├── uses AsyncSession (our Supabase repos)
├── uses models/ internally (maps rows ↔ entities)
├── converts models/ → schemas/ before returning
├── raises core/exceptions.py on failure
└── returns schemas/ upward

middleware/request_handler.py
├── logs every request (automatic)
└── catches core/exceptions.py that controllers missed (safety net)
```

### Model ↔ Schema Translation (inside repositories only)

**Reading from DB:** Repository executes SQLAlchemy query → gets model objects (full table shape) → converts to Pydantic schema (only fields the service needs) → returned upward.

**Writing to DB:** Repository receives Pydantic schema from service → creates SQLAlchemy model → `session.add()` + `session.commit()` → written to DB.

Example: pgvector returns rows via SQLAlchemy query. The repository gets `ManualChunk` SQLAlchemy model objects (all fields including embedding). Then converts to `KBSearchResult` Pydantic schemas (only `source`, `section`, `content`, `similarity`). The embedding vector and internal ID never leave the repository. `source` is the Zendesk article `html_url`; `section` is the Zendesk section name.

## 5. Exception Flow

### The Rule

```
Repository  → RAISES the error    (something broke at the data level)
Service     → IGNORES it          (lets it pass through)
Controller  → CATCHES it          (decides HTTP code + user-facing message)
Middleware  → CATCHES leftovers   (safety net for anything controllers missed)
```

### The Hierarchy

All defined in `core/exceptions.py`:

```
AppError (base)
├── IngestionError          legacy ingestion failures (see `core/exceptions.py`)
├── ZendeskAPIError         Zendesk Support API failures during ticketing / sync
└── …                       additional subclasses as defined in `core/exceptions.py`
```

`middleware/request_handler.py` catches `AppError` (the base class) — so ANY exception that a controller forgets to handle still gets caught and returned as a clean JSON error instead of a raw Python traceback.

### Why Services Ignore Exceptions

The service has no idea what HTTP code to return — HTTP is not its world. If the Zendesk Support API returns an error, `ZendeskAPIError` is raised from `zendesk_ticket_service` and handled in the chat stream / controller layer with a user-visible failure message.

The only time a service catches an exception is when it needs to make a business decision (for example, degrading gracefully when an optional subsystem fails while still returning a partial response).

## 7. Full Traced Example — Unresolved Escalation (Hardware or Software)

A user already logged into Infleet's portal types an unresolved issue such as "My GPS tracker screen is cracked" or "I'm getting error code E-4012 when connecting to the fleet dashboard."

### Middleware (automatic)

`request_handler.py` intercepts the request. Assigns correlation ID `req-7f3a`, logs "POST /chat arrived", starts timer. `cors.py` adds cross-origin headers. Request continues downward.

### Router — chat_router.py

Sees `POST /chat`, maps to `ChatController.handle_chat`. Calls `dependencies.py` to build the controller with all dependencies. Does nothing else.

### Controller — chat_controller.py

Receives `ChatRequest` schema. Validates `conversation_id` if provided. Calls `chat_service.handle_message_stream(request)` and returns a `StreamingResponse` with `media_type="text/event-stream"`. The controller does not return a `ChatResponse` — the chat service yields SSE events (`sources`, `chunk`, `done`) instead.

### Service — chat_service.py

Receives the message and conversation ID. Saves the user message to DB. If this is the first user turn in the conversation, skips KB search and calls OpenAI Chat Completions API (gpt-4o-mini) directly with the system prompt. For all subsequent turns, runs KB search first.

Yields `sources` SSE event with KB references, then calls OpenAI Chat Completions API (gpt-4o-mini) and yields each token as a `chunk` SSE event as it arrives.

### Repository — kb_repository.py

Generates embedding via OpenAI `text-embedding-3-small`. Runs pgvector similarity search against `manuals` table via SQLAlchemy `AsyncSession`. Gets SQLAlchemy model objects → converts to `KBSearchResult` schemas (strips embedding vector + internal ID) → returns schemas to service.

### Service — chat_service.py (continued)

After the OpenAI stream ends, the service inspects the accumulated response for a `[CREATE_TICKET]...[/CREATE_TICKET]` block. If found:

- Strips the block from the message text
- Parses the structured ticket fields (issue type, severity, device serial if applicable — `device_serial` is `null` for software/dashboard/login escalations)
- Calls `zendesk_ticket_service.create_ticket(...)`, then `ticket_service.save_ticket_record(...)` to persist the `tickets` row

### Service — zendesk_ticket_service.py

Posts to Zendesk Support `POST /api/v2/tickets.json` via httpx (Basic auth). Returns `ZendeskTicketResponse` (`zendesk_ticket_id`, `zendesk_ticket_url`). Device serial (when present) is appended to the ticket comment body for agents.

### Repository — ticket_repository.py

Persists `Ticket` SQLAlchemy rows (`ticket_id`, `conversation_id`, `issue_type`, `severity`, `device_serial`) after a successful external ticket create. Invoked via `TicketService.save_ticket_record` on the chat escalation path once Zendesk returns a ticket id.

### Service — chat_service.py (final)

Appends ticket confirmation to the final message text. Saves the assistant message to DB via `conversation_repository` (receives schema → creates SQLAlchemy model → `session.add()` + `session.commit()` to Supabase). Fires the terminal log block (KB times, OpenAI time, total response time).

Yields `done` SSE event with `{ type, confidence_tier, message, sources, ticket }` to the frontend.

### Controller — chat_controller.py

The `StreamingResponse` is already streaming — no further action required. If an exception escapes the service, the controller catches it and either lets the SSE error event propagate or returns an appropriate HTTP error before the stream starts.

### Middleware (automatic)

`request_handler.py` logs: "POST /chat completed — 200 — 1.2s — req-7f3a."

### What the user sees

> "I've created a support ticket for this unresolved issue. Your ticket number is 12345. A human agent will follow up in Zendesk."

### If something had failed

If the Zendesk Support API fails during ticket creation, `zendesk_ticket_service` raises `ZendeskAPIError`. `chat_service` surfaces a polite "ticketing system unavailable" style message on the chat path (no half-created ticket row).

Unhandled `AppError` subclasses still bubble to `middleware/request_handler.py`, which returns HTTP 500 with a generic message while logging details internally.

## Implementation Patterns — How Each Layer Is Coded

### Stack Rules

| Concern | Implementation | NOT used |
|---|---|---|
| **ORM** | SQLAlchemy 2.0 async (declarative models) | No Tortoise, no raw SQL for our DB |
| **DB driver** | `asyncpg` underneath SQLAlchemy async engine | No `psycopg2`, no sync drivers |
| **Migrations** | Alembic (auto-generates from SQLAlchemy models) | No manual SQL migrations |
| **DB sessions** | `AsyncSession` via `async_sessionmaker` | No raw pool for our Supabase DB |
| **HTTP client** | `httpx.AsyncClient` (Zendesk Help Center + Zendesk Support) | No `requests`, no `aiohttp` |
| **Schemas / DTOs** | `pydantic.BaseModel` | No dataclasses, no TypedDict |
| **Models / Entities** | SQLAlchemy `DeclarativeBase` (our DB) | No Pydantic for DB models |
| **Config** | `pydantic-settings.BaseSettings` | No `os.environ` scattered in code |
| **DI** | FastAPI `Depends()` factories in `dependencies.py` | No DI framework, no global singletons |
| **AI client** | `openai` Python SDK (async) | No LangChain, no LlamaIndex |
| **Embedding model** | `text-embedding-3-small` → `vector(1536)` locked | No `text-embedding-3-large` |
| **pgvector** | `pgvector.sqlalchemy` extension for SQLAlchemy | No raw vector SQL for our DB |
| **Async** | `async/await` everywhere — all routes, all repos, all services | No sync code in the request path |

### Database Strategy

Our Supabase DB uses **SQLAlchemy ORM** — we own the schema, we write models, Alembic generates migrations from those models, and repositories use `AsyncSession`.

### Layer Patterns

**Router** — Thin. One line maps URL to controller. Uses `Depends()` to get the controller.

```python
router = APIRouter()

@router.post("/chat")
async def chat(request: ChatRequest, ctrl: ChatController = Depends(get_chat_controller)):
    return await ctrl.handle_chat(request)  # returns StreamingResponse (text/event-stream)
```

**Controller** — Receives schema, calls service, catches exceptions, returns HTTP codes. Is a plain class, NOT decorated with `@router`.

```python
class ChatController:
    def __init__(self, chat_service: ChatService):
        self.chat_service = chat_service

    async def handle_chat(self, request: ChatRequest) -> StreamingResponse:
        try:
            return StreamingResponse(
                self.chat_service.handle_message_stream(request.message, request.conversation_id),
                media_type="text/event-stream",
                headers={"Cache-Control": "no-cache", "Connection": "keep-alive", "X-Accel-Buffering": "no"},
            )
        except AIServiceError:
            raise HTTPException(status_code=502, detail="AI service unavailable")
```

**Service** — Receives schemas from controller, calls repos, returns schemas. Constructor receives repos via DI. Never imports HTTPException.

```python
class ChatService:
    def __init__(self, kb_repo: KBRepository,
                 ticket_repo: TicketRepository,

                 openai_client: AsyncOpenAI):

        self.ticket_repo = ticket_repo
        self.conversations = conversation_repo
        self.openai = openai_client

        ...
```

**Repository (our DB — SQLAlchemy)** — Receives `AsyncSession` via DI. Uses ORM queries. Maps SQLAlchemy models to schemas before returning. Raises custom exceptions on failure.

```python
class KBRepository:
    def __init__(self, session: AsyncSession):
        self.session = session

    async def search(self, query_embedding: list[float], category: str) -> list[KBSearchResult]:
        try:
            stmt = select(ManualChunk).where(
                ManualChunk.category == category
            ).order_by(
                ManualChunk.embedding.cosine_distance(query_embedding)
            ).limit(5)
            result = await self.session.execute(stmt)
            rows = result.scalars().all()
            # SQLAlchemy model → Pydantic schema (strips embedding, strips id)
            return [KBSearchResult(source=r.source, section=r.section,
                                   content=r.content, similarity=r.similarity) for r in rows]
        except Exception as e:
            raise KBSearchError(f"pgvector search failed: {e}")
```

**Schema (DTO)** — Pydantic BaseModel. Used by all layers.

```python
class ChatRequest(BaseModel):
    user_id: str
    user_email: str
    device_serial: str | None = None  # collected by AI during conversation (not injected via page variables)
    message: str
    thread_id: str | None = None
```

```python
class TicketCreateRequest(BaseModel):
    user_id: str
    user_email: str
    device_serial: str | None = None  # NULL for software/dashboard/login escalations without a device
    issue_type: str
    severity: str
    ai_summary: str
```

**Model (Entity)** — SQLAlchemy declarative model. Used by repositories only. Mirrors our DB tables. Alembic reads these to generate migrations.

```python
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column
from pgvector.sqlalchemy import Vector

class Base(DeclarativeBase):
    pass

class ManualChunk(Base):
    __tablename__ = "manuals"

    id: Mapped[int] = mapped_column(primary_key=True)
    source: Mapped[str]
    section: Mapped[str]
    content: Mapped[str]
    category: Mapped[str]
    embedding = mapped_column(Vector(1536))  # pgvector — never leaves the repository
```

**db/supabase_pool.py** — Creates SQLAlchemy async engine + session factory. NOT a raw pool.

```python
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker

engine = create_async_engine(settings.supabase_db_url, echo=False)
async_session = async_sessionmaker(engine, expire_on_commit=False)
```

**dependencies.py** — Factory functions that build the chain. Provides `AsyncSession` for our repos.

```python
async def get_supabase_session() -> AsyncGenerator[AsyncSession, None]:
    async with async_session() as session:
        yield session

async def get_kb_repo(session: AsyncSession = Depends(get_supabase_session)) -> KBRepository:
    return KBRepository(session)

async def get_chat_service(...) -> ChatService:
    return ChatService(kb_repo=..., ticket_repo=..., conversation_repo=..., openai_client=...)

async def get_chat_controller(...) -> ChatController:
    return ChatController(chat_service=...)
```

**main.py** — App factory. Creates SQLAlchemy engine at startup, disposes on shutdown.

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.http_client = httpx.AsyncClient(timeout=30.0)
    yield
    await app.state.http_client.aclose()
    await engine.dispose()

app = FastAPI(lifespan=lifespan)
app.include_router(chat_router.router)
```

### Migrations Workflow

Models define the schema. Alembic reads the models and generates migration files. You run the migration and it creates/updates tables in Supabase.

```
1. You write/update a model in models/
2. Run: alembic revision --autogenerate -m "add conversations table"
3. Alembic compares models vs current DB → generates migration in db/migrations/versions/
4. Run: alembic upgrade head
5. Supabase DB is updated — table created or altered
```

You write code, not SQL. Alembic handles the SQL generation. Migration files are version-controlled in Git so every environment gets the same schema.

### Pattern Rule

When building any new component, follow this exact sequence: **Schema → Model (SQLAlchemy, if DB-backed) → run Alembic migration → Repository → Service → Controller → Router → wire in dependencies.py**. Copy the patterns above, swap the names.
