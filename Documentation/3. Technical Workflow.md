# Support Issue Workflow — Mapped to Production Stack (UPDATED)

> Every step below shows **exactly which tool** executes it, **how data moves** between tools, and **what format** the data travels in.

---

## Architecture Reference

| Role | Tool |
|---|---|
| Authentication | **Infleet's existing portal (handled by client)** |
| User Context Handoff | **Infleet session → React widget props** |
| Chat Interface | **Custom React Chat Component** *(Phase 1)* |
| Voice Interface | **Custom React Voice Widget** *(Phase 2)* |
| Voice Transport | **OpenAI Realtime API via WebSocket** *(Phase 2)* |
| AI Brain | **OpenAI Assistants API (direct)** |
| Knowledge Base | **pgvector (PostgreSQL)** — content sourced from Zendesk Help Center API |
| Backend Logic | **Python + FastAPI** |

## Step 1: The User Gateway *(Phase 1 — Chat / Phase 2 — Voice)*

```
User already logged in to Infleet Portal → Infleet injects user context → React Widget mounts → Selection screen
```

1. The user is already authenticated inside **Infleet's portal**. Login is fully handled by Infleet's existing system — we do not build, touch, or replicate it.

2. When the support widget loads inside the portal, Infleet's platform passes the logged-in user's identity to our React component as props or injected page variables:
   ```typescript
  // Infleet injects this into the page after their own login — user identity only.
  // Device identification is handled by the AI during conversation.
   window.__INFLEET_USER__ = {
     userId: "infleet_user_abc123",
    email: "john@example.com"
   };
   ```

3. Our **React Widget** reads this context on mount — no login UI, no auth logic on our side:
   ```typescript
   function ChatWidget() {
     // We consume Infleet's session — we don't create one
     const user = window.__INFLEET_USER__;
     const userId = user?.userId;
     const userEmail = user?.email;

     return (
       <Chat
         userId={userId}
         userEmail={userEmail}
       />
     );
   }
   ```

4. **Before any conversation begins, the widget displays a selection screen.** The user sees two options:

   > 💬 **Chat** — Type your question
   > 🎙️ **Voice** — Speak your question *(Phase 2)*

   - If the user chooses **Chat** → the existing chat flow continues exactly as described below (Step 2 onwards).
   - If the user chooses **Voice** → see **Step 1b** below.

   > **Scope note for Estimation:** The selection screen UI is not a separate milestone — it is included within the Chat UI milestone (Phase 1). It is a single React component rendered before the chat input mounts. Sara and Lind should treat it as part of the Chat UI build, not as surprise additional scope.

## Step 1b: Voice Path Entry *(Phase 2)*

```
User selects Voice → Browser requests mic → AudioWorklet → WebSocket → /voice-relay → OpenAI Realtime API → Audio response back to user
```

1. The browser requests microphone access via the Web Audio API:
   ```typescript
   // Voice path — browser side
   const stream = await navigator.mediaDevices.getUserMedia({ audio: true });
   const audioContext = new AudioContext({ sampleRate: 24000 });
   const source = audioContext.createMediaStreamSource(stream);

   // AudioWorklet encodes mic input as PCM16 and sends over WebSocket
   const ws = new WebSocket(`${BACKEND_WS_URL}/voice-relay`);
   ws.onmessage = (event) => {
     // Play incoming audio response from OpenAI back to user
     playAudioChunk(event.data, audioContext);
   };
   ```

2. The browser captures audio via **AudioWorklet**, which encodes the microphone input as **PCM16 frames** in real time.

3. PCM16 frames stream continuously over **WebSocket** to our **`/voice-relay` backend function**.

4. The **`/voice-relay` backend function** acts as a relay — it forwards the audio stream to the **OpenAI Realtime API** over its own WebSocket connection.

5. The **OpenAI Realtime API** handles the full pipeline in a single session:
   - **Speech-to-text:** Converts the incoming audio to text
   - **Reasoning:** Classifies intent, searches the knowledge base, determines the protocol
   - **Text-to-speech:** Converts the response back to audio

6. The audio response streams back through the `/voice-relay` to the browser in real time.

7. The browser plays the response audio via **AudioContext** — the user hears the assistant speak.

> **Data handoff:** The user's spoken question has been understood, classified, and the structured intent + protocol has been produced — equivalent to the output of Step 2 in the chat path. Both paths now converge at Step 2.

## Step 2: The Brain & Knowledge Retrieval (DECISION POINT)

```
POST /chat → FastAPI backend → KB search (pgvector) → OpenAI gpt-4o-mini → SSE stream to frontend
  ✅ Answer found in KB → Stream answer tokens → done event (go to Step 4)
  ❌ No answer found → Stream answer + [CREATE_TICKET] block → ticket created post-stream → done event with ticket data (go to Step 3)
```

> **Note — Voice path:** When the voice path (Step 1b) is active, intent classification and knowledge base search happen **inside the OpenAI Realtime API session** — not via a separate Assistants API call. The output is the same structured decision. Steps 3 and 4 are identical regardless of path.

> **Delivery method change:** `POST /chat` now returns a `StreamingResponse` with `media_type="text/event-stream"` instead of a JSON `ChatResponse`. The frontend reads the stream via `fetch()` + `res.body.getReader()` + `TextDecoder`, parses SSE events split on `\n\n`, and handles three event types: `sources`, `chunk`, `done`. The backend logic (KB search, confidence tiers, ticket creation, logging) is unchanged — only the delivery method changed from JSON to SSE.

1. The **`POST /chat` backend route** receives `{ message, conversation_id }` from the frontend and returns a `StreamingResponse` (`text/event-stream`). Response headers include `Cache-Control: no-cache`, `Connection: keep-alive`, and `X-Accel-Buffering: no` to prevent proxy buffering.

2. The backend saves the user message, checks conversation history, and determines whether to run KB search:
   - **First message in conversation (no prior user turns):** KB search is skipped — the system prompt and user message are sent directly to OpenAI. This is the greeting path.
   - **All subsequent messages:** KB search runs as normal (steps 3–6 below).

3. The user's message is embedded via **`text-embedding-3-small` (`vector(1536)`)** — locked across all ingestion and retrieval code.

4. **pgvector** runs a cosine similarity search against the `manuals` table (top 5 results). The `manuals` table is populated from Zendesk Help Center articles (German):
   ```sql
   -- embedding column type: vector(1536) — text-embedding-3-small, locked
   SELECT id, source, section, content, 1 - (embedding <=> $1) AS similarity
   FROM manuals
   ORDER BY embedding <=> $1
   LIMIT 5;
   ```

5. The backend applies **confidence tiers** based on the top similarity score and yields a `sources` SSE event:
   ```
   data: {"type": "sources", "sources": ["https://hilfe.infleet.de/hc/de/articles/123-Title — Section: Hardware-Anleitungen"]}
   ```

6. **🔀 THE CONFIDENCE TIER DECIDES HERE:**

   ### SCENARIO A: Answer Found in KB ✅ (HIGH or LOW tier)

   The backend sends the KB context + conversation history to **OpenAI `gpt-4o-mini`** and streams each token as a `chunk` SSE event:
   ```
   data: {"type": "chunk", "content": "To "}
   data: {"type": "chunk", "content": "reset "}
   data: {"type": "chunk", "content": "your GPS tracker..."}
   ```
   After the stream ends, the backend saves the assistant message to DB and yields the `done` event:
   ```
   data: {"type": "done", "confidence_tier": "high", "message": "...", "sources": [...], "ticket": null}
   ```
   **→ Flow goes directly to Step 4 (Output) — No escalation, no support ticket**

   ### SCENARIO B: No Answer Found — Escalation Required ❌ (NONE tier or AI includes `[CREATE_TICKET]` block)

   The AI streams its response tokens normally. If the accumulated response contains a `[CREATE_TICKET]...[/CREATE_TICKET]` block, the backend:
   - Strips the block from the final message
   - Parses the structured ticket fields (issue type, severity, device serial if applicable)
   - Creates the Zendesk support ticket via `httpx` after the stream ends (and persists a `tickets` row with `ticket_id`)
   - Appends the ticket confirmation to the final message
   - Yields the `done` event with `ticket` populated:
   ```
   data: {"type": "done", "confidence_tier": "none", "message": "Your support ticket has been created. Ticket number: 12345...", "sources": [], "ticket": {"ticket_id": "12345", "ticket_url": "https://{subdomain}.zendesk.com/agent/tickets/12345"}}
   ```
   **→ Flow proceeds to Step 3 (Ticket Creation + Zendesk)**

7. The frontend reads the SSE stream:
   ```typescript
   const res = await fetch(`${API_BASE}/chat`, {
     method: 'POST',
     headers: { 'Content-Type': 'application/json', Accept: 'text/event-stream' },
     body: JSON.stringify({ message, conversation_id }),
   });
   const reader = res.body!.getReader();
   const decoder = new TextDecoder();
   // Parse \n\n-delimited SSE blocks; dispatch on event.type
   ```

> **Data handoff:** Frontend receives a live token stream and the final `done` payload. Backend knows whether to include ticket data in `done`.

## Step 3: The Backend Action (Logic & Integrations) — ONLY IF NO FAQ ANSWER

```
/chat → Zendesk Support API → Human agent (Zendesk)
```

> **⚠️ This step ONLY executes if Step 2 returned `answer_found: false`**
>
> **Purpose:** When the AI cannot resolve the issue from the knowledge base, a Zendesk support ticket is created so a human agent can handle the case in Zendesk.
>
> Device identification (device serial/vehicle name) is handled by the AI during conversation when the issue is hardware-related; for software/dashboard/login escalations, `device_serial` may be omitted or sent as `null`.

1. The `/chat` backend service parses the `[CREATE_TICKET]` block and calls the Zendesk Support API in-process (same request), using Infleet user context plus AI-structured fields (`summary`, `description`, `issue_type`, `severity`, optional `device_serial`).

   **Hardware escalation — example payload** (`device_serial` populated):
   ```json
   {
     "user_id": "infleet_user_abc123",
     "user_email": "john@example.com",
     "device_serial": "GPS-TRK-00482",
     "issue_type": "hardware_failure",
     "sub_type": "cracked_screen",
     "severity": "high",
     "ai_summary": "Knowledge base did not provide a direct answer to the user's issue."
   }
   ```

   **Software escalation — example payload** (`device_serial` null):
   ```json
   {
     "user_id": "infleet_user_abc123",
     "user_email": "john@example.com",
     "device_serial": null,
     "issue_type": "software_issue",
     "sub_type": "fleet_dashboard_connection_error",
     "severity": "medium",
     "ai_summary": "Knowledge base did not provide a direct answer to the user's issue."
   }
   ```

2. **`ZendeskTicketService`** posts to `POST https://{subdomain}.zendesk.com/api/v2/tickets.json` via `httpx` with Basic auth (`{email}/token:{api_token}`). Payload shape (conceptual):

   ```json
   {
     "ticket": {
       "subject": "<AI summary line>",
       "comment": { "body": "<AI description + optional Device serial number line>" },
       "priority": "<severity mapped for Zendesk>",
       "tags": ["<issue_type>"]
     }
   }
   ```

3. Zendesk returns the numeric ticket `id` (e.g. `12345`) and the agent UI URL.

4. **The ticket is visible in Zendesk Support** for the human queue; our `tickets` table stores `ticket_id`, `issue_type`, `severity`, and optional `device_serial` for linking back to the conversation.

5. `/chat` yields the final `done` SSE payload (and the user sees the confirmation in the streamed assistant message).

> **Data handoff:** React component receives `ticket.ticket_id` and `ticket.ticket_url` in the `done` event. Proceeds to Step 4.

## Step 4: The Output

```
/chat backend function → React Component → User sees (or hears) response → Conversation saved
```

1. The React component delivers the final response based on the path taken:

   ### PATH A: FAQ Answer (No Escalation)
   
   **Chat path:** The React Chat Component renders:
   
   > ℹ️ *"To reset your GPS tracker, press and hold the power button for 10 seconds until the LED flashes red. Then release the button and wait for the device to reboot (approximately 30 seconds).*
   >
   > *Source: https://hilfe.infleet.de/hc/de/articles/... — Section: Hardware-Anleitungen*
   >
   > *Did this solve your issue?"*

   **Voice path *(Phase 2)*:** The OpenAI Realtime API speaks the same answer, with the source displayed as text.

   ### PATH B: Ticket Created (Escalation)

   **Chat path:** The React Chat Component renders:

   > ✅ *"I've created a support ticket for this unresolved issue.*
   >
   > *Your ticket number is **12345** (Zendesk).*
   >
   > *A human support agent will follow up with you shortly. You'll receive a confirmation email at john@example.com.*
   >
   > *Is there anything else I can help you with?"*

   **Voice path *(Phase 2)*:** The OpenAI Realtime API converts the same response to speech and the user **hears** it through their browser audio output. The ticket number is also displayed in the React Voice Widget as a text confirmation.

2. On conversation end, the full conversation history is saved to our **PostgreSQL database**.

## Key Change Summary

**BEFORE:** AI gated escalation behind an external eligibility check and created tickets only when that check succeeded.

**NOW:** 
1. AI checks vector database first
2. **IF answer found in FAQ/manuals** → Return answer immediately ✅
3. **ELSE (no answer found)** → Create Zendesk support ticket ❌

This prevents unnecessary ticket creation for questions that can be answered from the knowledge base.
