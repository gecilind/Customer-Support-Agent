# Voice Agent — Phase 2 Implementation Guide

---

## Overview

Add a voice conversation mode to the Infleet AI Support Agent. The user speaks naturally, the AI responds with voice. Same brain, same KB, same escalation, same ticket creation — just voice in, voice out.

**Transport:** OpenAI Realtime API — handles STT + reasoning + TTS in a single WebSocket session. No separate Whisper, no separate TTS service, no Silero VAD, no client-side ML.

**Conversation Style:** Turn-based (half-duplex with auto turn detection). User talks → pauses → AI processes → AI responds → mic reopens. No press-to-talk button. No interruption/barge-in.

## Workflow

```
Step 1: User clicks "Voice Agent" on selection screen
         ↓
Step 2: Browser requests microphone permission (getUserMedia)
         ↓
Step 3: AudioWorklet starts capturing PCM16 16kHz mono frames
         ↓
Step 4: Frontend opens WebSocket to /voice-relay
         ↓
Step 5: /voice-relay opens WebSocket to OpenAI Realtime API
         ↓
Step 6: Session configured with VAD settings (see below)
         ↓
Step 7: AI sends greeting audio → browser plays through speaker
         ↓
Step 8: User talks → PCM16 frames stream continuously to OpenAI
         ↓
Step 9: User stops talking → 1.5s silence detected by server VAD
         ↓
Step 10: OpenAI transcribes audio → text
         ↓
Step 11: voice_service calls chat_service.handle_message(text)
          → same KB search, same confidence tiers, same escalation
         ↓
Step 12: Response text sent to Realtime API → TTS audio generated
         ↓
Step 13: Audio frames stream back → browser plays via AudioContext
         ↓
Step 14: AI finishes speaking → mic reopens → repeat from Step 8
```

## OpenAI Realtime API Session Configuration

```json
{
  "model": "gpt-4o-realtime-preview",
  "modalities": ["text", "audio"],
  "voice": "nova",
  "turn_detection": {
    "type": "server_vad",
    "threshold": 0.7,
    "silence_duration_ms": 1500,
    "prefix_padding_ms": 300
  },
  "input_audio_format": "pcm16",
  "output_audio_format": "pcm16",
  "input_audio_transcription": {
    "model": "whisper-1"
  }
}
```

> **Voice options:** `alloy`, `echo`, `fable`, `onyx`, `nova`, `shimmer`. We use `nova` as default — professional and neutral. Can be changed in config to match Infleet's brand preference. Test different voices with Sara during development.
```

## Noise Detection & VAD Configuration

### What Each Setting Does

| Setting | Value | Purpose |
|---|---|---|
| `type` | `server_vad` | OpenAI handles all voice activity detection server-side. No client-side ML needed. |
| `threshold` | `0.7` | Confidence level required to detect human speech. Higher = stricter. 0.7 filters out most background noise (TV, traffic, keyboard). Default is 0.5 — we raise it for noisy environments. |
| `silence_duration_ms` | `1500` | How long the user must be silent before OpenAI considers them "done talking." 1500ms (1.5 seconds) is a natural pause. Too short (500ms) = cuts off mid-thought. Too long (3000ms) = feels laggy. |
| `prefix_padding_ms` | `300` | Captures 300ms of audio BEFORE speech was detected. Prevents clipping the first syllable. |

### STT Failsafe (Noise Rejection)

Even with VAD threshold 0.7, loud noises can occasionally pass through (door slam, cough, dog bark). The failsafe:

1. OpenAI transcribes the audio
2. If transcription is empty, whitespace only, or just punctuation ("...", ",", ".")
3. Backend ignores it — does NOT call `chat_service.handle_message()`
4. Mic stays open, keeps listening
5. User doesn't notice anything

This catches 95%+ of false triggers without any client-side ML.

### Tunable Values

These can be adjusted based on testing:

| Setting | Range | When to Change |
|---|---|---|
| `threshold` | 0.5 — 0.9 | If too many false triggers in noisy environments → increase. If AI doesn't hear soft-spoken users → decrease. |
| `silence_duration_ms` | 800 — 2500 | If AI cuts off users mid-sentence → increase. If response feels too slow → decrease. |
| `prefix_padding_ms` | 100 — 500 | If first word is clipped → increase. |

## Audio Pipeline Detail

### Browser → Backend (User's Voice)

```
Microphone (44.1kHz/48kHz Float32)
  → AudioWorklet (pcm-worklet.ts)
  → Downsample to 16kHz mono
  → Convert Float32 to Int16 (PCM16)
  → WebSocket binary frames to /voice-relay
  → Relay to OpenAI Realtime API
```

### Backend → Browser (AI's Voice)

```
OpenAI Realtime API generates TTS audio (PCM16 24kHz)
  → WebSocket binary frames to /voice-relay
  → Relay to browser WebSocket
  → AudioContext decodes PCM16
  → Plays through speaker
```

## What the Voice Path Shares with Chat

| Component | Shared? | Notes |
|---|---|---|
| System prompt | YES | Same `SYSTEM_PROMPT` in `chat_service.py` |
| KB search (pgvector) | YES | Same `kb_service.search()` |
| Query reformulation | YES | Same `_reformulate_query()` |
| Confidence tiers | YES | Same HIGH/LOW/NONE thresholds |
| Escalation + ticket creation | YES | Same `[CREATE_TICKET]` flow |
| Conversation history | YES | Same `messages` table, same `conversations` table |
| Jira integration | YES | Same `/create-ticket` route internally |

The only difference: voice responses should be shorter and more conversational. Long numbered lists sound robotic when spoken aloud. **This is a required prompt change.**

In `voice_service.py`, when calling `chat_service.handle_message()`, pass a flag indicating the request is from voice. Then in `chat_service.py`, when building the OpenAI messages, prepend this instruction to the user message if the request is from voice:

```
VOICE MODE: The user is communicating via voice. Keep your response concise and conversational — under 3-4 sentences when possible. Do NOT list 10 numbered steps. Summarize the key action briefly and offer to provide more detail if needed. Speak naturally as if you are on a phone call, not writing a document.
```

**How to implement this:**

1. Add an optional `is_voice: bool = False` parameter to `chat_service.handle_message()`
2. `voice_service.py` calls `handle_message(text, conversation_id, is_voice=True)`
3. Chat route calls `handle_message(text, conversation_id)` — defaults to `False`, no change
4. In `handle_message`, if `is_voice` is True, prepend the VOICE MODE instruction to the message before sending to OpenAI
5. The KB search still uses the original message — VOICE MODE instruction is only for response generation

**This does NOT change the system prompt.** It's a per-request instruction that only applies to voice calls. The chatbot continues to give detailed numbered lists as before.

## Architecture Rule Reminder

```
Browser (VoiceInterface + AudioWorklet)
  ↕ WebSocket (PCM16 binary frames)
Backend (/voice-relay → voice_service)
  ↕ WebSocket (PCM16 binary frames)
OpenAI Realtime API (VAD + STT + TTS)

voice_service → chat_service.handle_message() → same pipeline as chat
```

No telephony. No phone numbers. No Twilio. No Amazon Connect. Browser-only, in-portal voice.
