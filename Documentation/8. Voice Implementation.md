# Voice Agent — Implementation (as built)

---

## Overview

Voice mode uses **two** OpenAI surfaces:

1. **OpenAI Realtime API (WebSocket)** — microphone path: server VAD, input audio buffer, **Whisper-1 transcription**, and **verbatim TTS** only. Session instructions tell the model it is a relay, not the support brain.
2. **Chat Completions (existing `chat_service`)** — every user utterance (after transcription) runs the **same** pipeline as web chat: KB search, tiers, escalation, ticket flow, DB messages. `handle_message(..., is_voice=True)` consumes the internal SSE stream and returns the final assistant text; `voice_service` then pushes that text into Realtime via `response.create` for speech.

**Transport:** Browser `WebSocket` to `WS /voice-relay` (FastAPI). Backend opens a second WebSocket to `wss://api.openai.com/v1/realtime?model={OPENAI_REALTIME_MODEL}` with `OpenAI-Beta: realtime=v1`. Binary frames are PCM16 uplink; downlink TTS is PCM16 chunks relayed as binary.

**Half-duplex by default:** Worklet mic capture is **muted** whenever mode is not `listening`. **Barge-in is implemented:** if the user speaks while the assistant is in `speaking`, the backend cancels the Realtime response and tells the frontend to flush queued playback.

## High-level flow

1. User opens Voice Agent → `getUserMedia` → `AudioContext` + `pcm-processor` worklet (fallback inline processor if worklet module load fails).
2. WebSocket to `${VITE_API_URL or http://localhost:8000}` with path `/voice-relay` (http→ws).
3. Backend connects to OpenAI Realtime, sends `session.update`, waits for `session.updated`.
4. **Greeting:** `chat_service.handle_message(message="hello", conversation_id=None, is_voice=True)`; assistant text sent to browser as JSON `transcript` (role `assistant`); then `_trigger_tts` issues Realtime `response.create` so the model **speaks that text verbatim**.
5. User audio: worklet posts PCM16 buffers → browser sends **binary** frames → backend base64-encodes → `input_audio_buffer.append`.
6. On `input_audio_buffer.speech_stopped` (and not in a cancel-only path), mode may move to `processing`; input buffer cleared with `input_audio_buffer.clear`.
7. On `conversation.item.input_audio_transcription.completed`, transcript is validated (non-empty, not noise-only regex); duplicates by `item_id` ignored; user `transcript` JSON to browser; `chat_service.handle_message(transcript, conversation_id, is_voice=True)`; assistant `transcript`; optional `ticket_created` with Jira URL; `_trigger_tts` for reply audio.
8. TTS: first `response.audio.delta` transitions mode to `speaking` and streams **binary** PCM to the browser. On `response.audio.done`, backend sends `tts_stream_ended`; it **waits** on an asyncio Event until the frontend sends JSON `{ "type": "playback_drained" }` after its playback queue drains; then mode → `listening` (with optional echo cooldown — see below).

## Realtime `session.update` (actual shape)

Sent in `_configure_session` after connect:

- **`modalities`:** `["text", "audio"]`
- **`voice`:** `"ash"` (TTS voice for the relay utterances)
- **`instructions`:** Relay persona — speak user-provided text **exactly**, English only; no reasoning for support content.
- **`turn_detection`:** `server_vad`, `threshold` **0.8**, `silence_duration_ms` **1500**, `prefix_padding_ms` **300**, **`create_response`: false** (responses are only created explicitly after `chat_service` returns text).
- **`input_audio_format` / `output_audio_format`:** `pcm16`
- **`input_audio_transcription`:** `model` `whisper-1`, **`language`:** `"en"`

## Browser ↔ `/voice-relay` protocol

**Client → server**

| Payload | Purpose |
|---|---|
| Binary `ArrayBuffer` | PCM16 mono 16 kHz mic frames (from worklet). |
| `{"type":"playback_drained","ts":"..."}` | Playback queue finished after `tts_stream_ended`; unblocks backend `speaking` → `listening`. |

**Server → client**

| `type` | Purpose |
|---|---|
| `mode_start` | `{ mode, ts }` — `listening` \| `processing` \| `speaking`. Drives UI and worklet mute. |
| `mode_end` | `{ mode, ts }` — logging / symmetry with start. |
| `transcript` | `{ role, text, conversation_id? }` — `user` or `assistant` lines for the on-screen transcript. |
| `ticket_created` | `{ url }` — Jira link after escalation. |
| `tts_stream_ended` | `{ ts }` — Realtime finished sending audio for this utterance; FE should drain sources then send `playback_drained`. The `ts` value is produced by `_now_ts()` in `voice_service.py` — that helper is used **only** for this payload. |
| `flush_audio` | Stop all scheduled `AudioBufferSourceNode`s, reset queue; used on barge-in. |
| `error` | `{ message }` — session-level failure. |

`VoiceService._transition` formats `mode_start` / `mode_end` timestamps with inline `datetime.now().strftime("%H:%M:%S.%f")[:-3]`, not `_now_ts()`.

## Mode machine (backend is source of truth)

- **`listening`:** User may speak; worklet **unmuted**.
- **`processing`:** Turn committed / thinking; worklet **muted**.
- **`speaking`:** TTS playing; worklet **muted**.

Transitions are implemented in `VoiceService._transition` (emits `mode_end` / `mode_start` with timestamps). When leaving **`speaking`** for **`listening`**, an **~0.8 s sleep** runs (“echo guard”) before the next `listening` starts, reducing speaker→mic bleed.

**Barge-in:** On `input_audio_buffer.speech_started` while `mode == "speaking"`, the service sets `cancelling`, sends Realtime `response.cancel`, sends browser `flush_audio`, and moves to `processing`.

---

## Transcript hygiene

- **Noise:** `_NOISE_PATTERN` — if stripped transcript is empty or only whitespace/punctuation (including `…`, em dash, etc.), it is **not** sent to `chat_service`.
- **Duplicates:** `last_transcribed_item_id` skips a second completion for the same `item_id`.

After a usable transcript or after `speech_stopped` in some paths, **`input_audio_buffer.clear`** runs so VAD does not reuse stale audio.

## Chat pipeline integration

- **`handle_message(message, conversation_id, *, is_voice=False)`** — async-for through `handle_message_stream`, parses terminal SSE `done` into `ChatResponse` (`conversation_id`, `message`, `sources`, `confidence_tier`, optional `jira_ticket_url`).
- **`VOICE_MODE_INSTRUCTION`** in `chat_service.py` — concise spoken replies; **explicitly** states escalation/ticket rules still apply; collection stays conversational but complete; `[CREATE_TICKET]` flow unchanged. When `is_voice` is true, it is prepended to:
  - **`none_user_content`** (NONE-tier path)
  - **`kb_user_content`** (RAG path)

The voice greeting runs through `handle_message` like any other turn and lands in whichever of those branches applies — there is **no** separate first-turn-only injection.

KB embedding query still uses the **raw** user message / reformulation logic without injecting that instruction into the embedding text.

## Frontend audio / UI notes

- **Capture:** Native rate → worklet → 16 kHz PCM16 (nearest-neighbour decimation); mute via `port.postMessage({ type: 'set_muted', value })` from main thread when mode ≠ `listening`.
- **Playback:** Incoming PCM16 interpreted as **24 kHz** mono (`createBuffer(1, len, 24000)`), chained `AudioBufferSourceNode`s scheduled with `nextPlayTimeRef`; **`playbackRate` 1.1** for slightly faster playback.
- **`trySendPlaybackDrained`:** Only sends when `tts_stream_ended` was received **and** `playbackPendingRef === 0`.
- **States:** `connecting` \| `listening` \| `processing` \| `speaking` \| `error` — copy hints half-duplex and “mic off until listening”.
- **Teardown:** Back or unmount closes WebSocket, disconnects worklet, stops tracks, suspends `AudioContext`.

## Architecture diagram

```
Browser (VoiceInterface + pcm-processor)
  ↔ WebSocket: JSON control + binary PCM16 up/down
VoiceController → VoiceService
  ↔ WebSocket: Realtime API (VAD, STT, verbatim TTS)
VoiceService → ChatService.handle_message(is_voice=True)
  → same KB / tiers / tickets / DB as POST /chat stream
```

No telephony stack; browser-only relay through your FastAPI backend.
