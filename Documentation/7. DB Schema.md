# Database Schema вҖ” Infleet AI Support Agent

---

## Overview

We own **one database** вҖ” Supabase PostgreSQL. This schema defines every table in it.

All tables below are created via **Alembic migrations** auto-generated from **SQLAlchemy models**. Developers write models in `models/`, run `alembic revision --autogenerate`, and Alembic produces the SQL. Nobody writes CREATE TABLE by hand.

---

## Tables

We have **4 tables**:

| Table | Purpose |
|---|---|
| `manuals` | Document chunks + embeddings for RAG knowledge base |
| `conversations` | Chat sessions вҖ” one per user interaction thread |
| `messages` | Individual messages within a conversation (user + assistant) |
| `tickets` | References to Zendesk support tickets created during conversations |

## Table: `manuals`

Stores chunked Zendesk Help Center article content with vector embeddings for RAG retrieval. Each row is one chunk of a Zendesk article (German). Content is populated via `POST /sync-zendesk` вҖ” not by file upload.

| Column | Type | Constraints | Why |
|---|---|---|---|
| `id` | `SERIAL` | `PRIMARY KEY` | Unique row identifier |
| `article_id` | `BIGINT` | `NOT NULL` | Zendesk article numeric ID вҖ” used to upsert (delete old chunks then insert new) when re-syncing a specific article |
| `source` | `VARCHAR(500)` | `NOT NULL` | Zendesk article `html_url` (e.g., `https://hilfe.infleet.de/hc/de/articles/123-Title`) вҖ” the AI cites this URL as the source of the answer |
| `section` | `VARCHAR(500)` | `NOT NULL` | Zendesk section `name` (e.g., "Hardware-Anleitungen", "Infleet", "Admin") вҖ” groups articles by their Zendesk section |
| `content` | `TEXT` | `NOT NULL` | Plain-text chunk extracted from the article HTML body вҖ” this is what the AI reads and returns to the user |
| `category` | `VARCHAR(100)` | `NOT NULL, DEFAULT 'general'` | Zendesk category `name` (e.g., "Hardware", "Software", "Allgemein", "News") вҖ” used as a WHERE filter in pgvector queries |
| `chunk_index` | `INTEGER` | `NOT NULL, DEFAULT 0` | Position of this chunk within the article вҖ” preserves reading order when an article produces multiple chunks |
| `embedding` | `VECTOR(1536)` | `NOT NULL` | Vector embedding generated by `text-embedding-3-small` вҖ” the column pgvector searches against. 1536 dimensions, locked to this model |
| `created_at` | `TIMESTAMPTZ` | `NOT NULL, DEFAULT NOW()` | When this chunk was synced вҖ” useful for tracking the last Zendesk sync |

### Indexes

| Index | Type | On | Why |
|---|---|---|---|
| `ix_manuals_embedding_hnsw` | `HNSW (vector_cosine_ops)` | `embedding` | **Required from Day 1.** Without this, pgvector does a full table scan on every query. HNSW gives approximate nearest neighbor search in milliseconds. Config: `m=16, ef_construction=64` |
| `ix_manuals_category` | `BTREE` | `category` | The pgvector query filters by category before searching embeddings. Without this index, Postgres scans every row to check the category string before the vector comparison |
| `ix_manuals_article_id` | `BTREE` | `article_id` | Used when re-syncing an article вҖ” delete all chunks WHERE article_id = {id} before inserting new chunks |

### SQL (for reference вҖ” Alembic generates this)

```sql
CREATE TABLE manuals (
    id SERIAL PRIMARY KEY,
    article_id BIGINT NOT NULL,
    source VARCHAR(500) NOT NULL,
    section VARCHAR(500) NOT NULL,
    content TEXT NOT NULL,
    category VARCHAR(100) NOT NULL DEFAULT 'general',
    chunk_index INTEGER NOT NULL DEFAULT 0,
    embedding VECTOR(1536) NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX ix_manuals_embedding_hnsw ON manuals
    USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);

CREATE INDEX ix_manuals_category ON manuals (category);
CREATE INDEX ix_manuals_article_id ON manuals (article_id);
```

## Table: `conversations`

Stores one row per conversation session. A conversation starts when the user sends their first message and persists across the entire thread.

| Column | Type | Constraints | Why |
|---|---|---|---|
| `id` | `SERIAL` | `PRIMARY KEY` | Internal unique identifier |
| `thread_id` | `VARCHAR(200)` | `NOT NULL, UNIQUE` | OpenAI Assistants API thread ID (e.g., "thread_abc123") вҖ” this is how we resume a conversation. Unique because one thread = one conversation |
| `user_id` | `VARCHAR(200)` | `NOT NULL` | Infleet's user identifier from `window.__INFLEET_USER__` вҖ” links the conversation to a specific Infleet user without storing their auth data |
| `user_email` | `VARCHAR(300)` | `NOT NULL` | User's email from Infleet context вҖ” stored so ticket creation and conversation history can reference it without re-fetching from Infleet |
| `device_serial` | `VARCHAR(200)` | `NULL` | Device serial collected by the AI during conversation вҖ” nullable because it is not known at conversation start and the user may be asking a general question not tied to a specific device |
| `status` | `VARCHAR(50)` | `NOT NULL, DEFAULT 'active'` | Conversation state: `active` (in progress), `resolved` (user confirmed fix), `escalated` (Zendesk support ticket created). Used for analytics and filtering |
| `created_at` | `TIMESTAMPTZ` | `NOT NULL, DEFAULT NOW()` | When the conversation started |
| `updated_at` | `TIMESTAMPTZ` | `NOT NULL, DEFAULT NOW()` | Last activity timestamp вҖ” updated on every new message. Used to find stale conversations |

### Indexes

| Index | Type | On | Why |
|---|---|---|---|
| `ix_conversations_thread_id` | `BTREE, UNIQUE` | `thread_id` | Every `/chat` request includes `thread_id` to resume the conversation вҖ” this lookup must be fast |
| `ix_conversations_user_id` | `BTREE` | `user_id` | Load conversation history for a specific user (e.g., "show me my past support sessions") |
| `ix_conversations_status` | `BTREE` | `status` | Filter active vs resolved vs escalated conversations for analytics dashboards |

### SQL (for reference вҖ” Alembic generates this)

```sql
CREATE TABLE conversations (
    id SERIAL PRIMARY KEY,
    thread_id VARCHAR(200) NOT NULL UNIQUE,
    user_id VARCHAR(200) NOT NULL,
    user_email VARCHAR(300) NOT NULL,
    device_serial VARCHAR(200),
    status VARCHAR(50) NOT NULL DEFAULT 'active',
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX ix_conversations_thread_id ON conversations (thread_id);
CREATE INDEX ix_conversations_user_id ON conversations (user_id);
CREATE INDEX ix_conversations_status ON conversations (status);
```

## Table: `messages`

Stores every individual message within a conversation вҖ” both user messages and assistant responses. This is the full chat log.

| Column | Type | Constraints | Why |
|---|---|---|---|
| `id` | `SERIAL` | `PRIMARY KEY` | Unique message identifier |
| `conversation_id` | `INTEGER` | `NOT NULL, FK вҶ’ conversations.id` | Links this message to its parent conversation. `ON DELETE CASCADE` вҖ” if a conversation is deleted, all its messages go with it |
| `role` | `VARCHAR(50)` | `NOT NULL` | Who sent this message: `user` or `assistant`. Matches OpenAI's message role format so conversation replay is straightforward |
| `content` | `TEXT` | `NOT NULL` | The actual message text. For user messages, this is what they typed. For assistant messages, this is the AI's response including any ticket numbers or instructions |
| `created_at` | `TIMESTAMPTZ` | `NOT NULL, DEFAULT NOW()` | When this message was sent вҖ” preserves conversation ordering and enables time-based queries |

### Indexes

| Index | Type | On | Why |
|---|---|---|---|
| `ix_messages_conversation_id` | `BTREE` | `conversation_id` | Every time we load a conversation, we fetch all its messages вҖ” this index makes that query fast |
| `ix_messages_created_at` | `BTREE` | `created_at` | Sort messages chronologically within a conversation. Also supports time-range queries for analytics |

### SQL (for reference вҖ” Alembic generates this)

```sql
CREATE TABLE messages (
    id SERIAL PRIMARY KEY,
    conversation_id INTEGER NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
    role VARCHAR(50) NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX ix_messages_conversation_id ON messages (conversation_id);
CREATE INDEX ix_messages_created_at ON messages (created_at);
```

## Table: `tickets`

Stores references to Zendesk support tickets created during conversations. Full ticket threads, assignees, and SLA live in Zendesk вҖ” we only persist identifiers and AI metadata for analytics and UI deep links.

| Column | Type | Constraints | Why |
|---|---|---|---|
| `id` | `SERIAL` | `PRIMARY KEY` | Internal unique identifier |
| `conversation_id` | `INTEGER` | `NOT NULL, FK вҶ’ conversations.id` | Links ticket to the conversation that triggered it. `ON DELETE CASCADE` |
| `ticket_id` | `VARCHAR(100)` | `NOT NULL, UNIQUE` | External ticket identifier (Zendesk numeric id as text, e.g. `"12345"`) вҖ” used for UI + GET `/conversations/{id}` without calling Zendesk again |
| `issue_type` | `VARCHAR(100)` | `NOT NULL` | Classification from the AI (e.g., "hardware_failure", "cracked_screen", "software_config_unresolved", "integration_issue") вҖ” any issue type the AI classified during escalation |
| `severity` | `VARCHAR(50)` | `NOT NULL, DEFAULT 'medium'` | Priority level the AI assigned: `low`, `medium`, `high`, `critical` вҖ” mapped to Zendesk ticket priority on create |
| `device_serial` | `VARCHAR(200)` | `NULL` | Which device this ticket is about вҖ” NULL for software/dashboard/login issues that aren't tied to a specific device. Populated for hardware issues where the AI collected the device serial during conversation. |
| `created_at` | `TIMESTAMPTZ` | `NOT NULL, DEFAULT NOW()` | When the ticket was created |

### Indexes

| Index | Type | On | Why |
|---|---|---|---|
| `ix_tickets_conversation_id` | `BTREE` | `conversation_id` | Look up tickets for a specific conversation |
| `ix_tickets_ticket_id` | `BTREE, UNIQUE` | `ticket_id` | Prevent duplicate ticket records. Also used if we need to look up a conversation by ticket number |
| `ix_tickets_device_serial` | `BTREE` | `device_serial` | Query "has this device had tickets before?" вҖ” useful for detecting repeat issues |

### SQL (for reference вҖ” Alembic generates this)

```sql
CREATE TABLE tickets (
    id SERIAL PRIMARY KEY,
    conversation_id INTEGER NOT NULL REFERENCES conversations(id) ON DELETE CASCADE,
    ticket_id VARCHAR(100) NOT NULL UNIQUE,
    issue_type VARCHAR(100) NOT NULL,
    severity VARCHAR(50) NOT NULL DEFAULT 'medium',
    device_serial VARCHAR(200),
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX ix_tickets_conversation_id ON tickets (conversation_id);
CREATE INDEX ix_tickets_ticket_id ON tickets (ticket_id);
CREATE INDEX ix_tickets_device_serial ON tickets (device_serial);
```

## What We Do NOT Store

| Data | Where It Lives | Why We Don't Store It |
|---|---|---|
| User authentication / passwords | Infleet's portal | We don't build auth вҖ” we consume their session |
| User profile details | Infleet's database | We only store `user_id` and `user_email` from their injected context. `device_serial` is collected by the AI during conversation and stored on the conversation and ticket records |
| Full Zendesk ticket thread | Zendesk | Zendesk is the source of truth for ticket status, assignee, macros, and comments. We only store `ticket_id` + AI metadata |
| OpenAI thread content | OpenAI | OpenAI stores the thread history on their side. We store our own copy in `messages` as a backup and for analytics |

## pgvector Notes

**Extension must be enabled before first migration:**

```sql
CREATE EXTENSION IF NOT EXISTS vector;
```

This is a one-time setup in Supabase (enabled via their dashboard or a manual SQL command). Alembic cannot auto-generate this вҖ” it must exist before the `manuals` table migration runs.

**Embedding model is locked:** `text-embedding-3-small` produces 1536-dimension vectors. The `VECTOR(1536)` column type, the HNSW index, and every embedding generation call must all use 1536. Switching to `text-embedding-3-large` (3072 dims) would require dropping the HNSW index, altering the column, re-generating every embedding, and rebuilding the index. Do not change the model without a full re-sync plan.

**HNSW index parameters:**
- `m = 16` вҖ” number of connections per node. Higher = better recall, more memory. 16 is the standard default.
- `ef_construction = 64` вҖ” build-time search breadth. Higher = better index quality, slower build. 64 is the standard default.

These values are correct for a knowledge base under 100k chunks. If the manual corpus grows past that, revisit both values.
